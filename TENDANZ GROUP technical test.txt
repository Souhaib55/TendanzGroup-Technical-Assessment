1. Contexte & Objectif 

Tendanz Group est un cabinet de conseil en data et technologie. Dans le cadre d’un projet 
client dans le secteur assurance, vous devez développer un moteur de tarification permettant 
de calculer le prix d’une couverture d’assurance en fonction du profil client, du produit choisi et 
de la zone géographique. 

Un squelette de projet vous est fourni avec la structure, les entités JPA, la base de données H2 
pré-remplie et les modèles Angular. Votre mission est d’implémenter la logique métier, les 
endpoints REST et l’interface utilisateur. 

2. Règles Métier 

2.1 Formule de tarification 

Prix Final = Taux de Base × Facteur Âge × Coefficient Zone 

Facteurs d’âge 

| Tranche d’âge   | Catégorie | Facteur |
|-----------------|-----------|---------|
| 18 - 24 ans     | YOUNG     | 1.30    |
| 25 - 45 ans     | ADULT     | 1.00    |
| 46 - 65 ans     | SENIOR    | 1.20    |
| 66 - 99 ans     | ELDERLY   | 1.50    |

Zones géographiques 

| Zone          | Code | Coefficient |
|---------------|------|-------------|
| Grand Tunis   | TUN  | 1.20        |
| Sfax          | SFX  | 1.00        |
| Sousse        | SOU  | 1.10        |

Produits d’assurance 

| Produit                | Code    | Taux de Base (TND) |
|------------------------|---------|--------------------|
| Assurance Auto         | AUTO    | 500.00             |
| Assurance Habitation   | HABITAT | 300.00             |
| Assurance Santé        | SANTE   | 800.00             |

Exemple : Client de 30 ans, zone Tunis, Assurance Auto = 500 × 1.00 × 1.20 = 600.00 TND 
 
 


3. Travail Demandé 

Vous devez compléter les fichiers marqués TODO dans le squelette fourni. 

3.1 Backend (Spring Boot) 

1. PricingService.calculateQuote() — Implémenter la logique de calcul du prix selon les 

règles métier ci-dessus. Gérer les cas d’erreur (produit inexistant, zone inconnue). 
2. QuoteController — 3 endpoints REST : POST /api/quotes (créer un devis), GET 

/api/quotes/{id} (détail), GET /api/quotes (liste avec filtres optionnels productId et 
minPrice). 
3. QuoteRepository — Méthodes custom : findByClientName, findByProductId, requête 

custom pour devis au-dessus d’un seuil. 
4. GlobalExceptionHandler — @ControllerAdvice avec gestion propre des exceptions (404, 

400, 500). 
5. Tests unitaires — Compléter PricingServiceTest avec au minimum 5 cas de test 

couvrant les différentes tranches d’âge et cas d’erreur. 

3.2 Frontend (Angular) 

1. QuoteService — Implémenter les appels HTTP vers l’API (createQuote, getQuote, 

getQuotes) avec gestion d’erreurs. 
2. Formulaire de devis — Reactive Form avec validations (champs requis, âge 18-99), 

dropdown produits chargé depuis l’API, soumission et affichage du résultat. 
3. Liste des devis — Tableau avec tri par date/prix, filtres par produit et prix minimum, 

navigation vers le détail. 
4. Détail du devis — Affichage complet avec décomposition du prix et règles appliquées. 

3.3 Bonus (non obligatoire) 

• 
Pagination côté backend et frontend 
• 
Export PDF du devis 
• 
Historique des modifications d’un devis 
• 
Ajout d’un 4ème produit avec ses règles spécifiques

4.1 Ce qu’on évalue 

• 
Qualité du code : lisibilité, nommage, structure, séparation des responsabilités 
• 
Compréhension des règles métier : la formule de pricing est-elle correctement 
implémentée ? 
• 
API REST : design des endpoints, codes HTTP, gestion d’erreurs, validation 
• 
Angular : utilisation des Reactive Forms, services, composants, routing 
• 
Tests : couverture, cas limites, pertinence des assertions 
• 
Historique Git : commits progressifs et clairs 

4.2 Ce qu’on ne cherche PAS 

• 
Un design pixel-perfect (un CSS propre et fonctionnel suffit) 
• 
De la sur-ingénierie (pas besoin de Kafka, Redis ou microservices) 
• 
Un projet parfait — on préfère un projet incomplet mais bien fait qu’un projet complet 
mais bricolé
